Solid spec. Better than most architecture docs — principles are clear, gates are tight, the derivation trace will save time in six months when you're wondering why a decision is the way it is. The main issues below are real but most are local fixes, not structural rewrites. One exception flagged.

## Blocking — must resolve before Day 1 scaffold code

1. **§8.3 contradicts §11 item 2 on weight-promotion human gating.** §8.3 step 5 says candidates are "presented to the human for approval"; §11 item 2 says "Controller-weight updates auto-promote; human sees audit log." These are directly opposed. The repo can't be frozen with the contradiction in place — pick one. My read is §11 is the intended position (auto-promotion with audit, rollback via superseding row) and §8.3 is stale text from the promotion-protocol round. Fix by rewriting §8.3 step 5 to match §11, and log as v1.1 revision in §0.

2. **`target_variable: str` is a silent-failure vector.** A typo (`wti_front_month_clsoe`) creates a new target that no Print will ever match; the Forecast persists, grading never fires, and the desk looks like it's emitting but contributes nothing to attribution. Two fixes, either works:
   - A `target_variables.py` module in `contracts/` with a frozen registry (`WTI_FRONT_MONTH_CLOSE = "wti_front_month_close"`), imported by every desk and the bus validator. String mismatches become import errors at module load, not runtime ghosts.
   - A `TargetVariable` enum in `contracts/v1.py`.

   The enum is cleaner but less extensible — new targets need a v1.x revision. The registry approach is more permissive but requires discipline. Given the portability target (equity VRP needs entirely new target names), registry is probably right.

3. **Horizon semantics are ambiguous.** `ClockHorizon` with `duration: timedelta` has no anchor. A Wednesday-emission 7-day-clock-horizon grades against next Wednesday's timestamp, but the Print it should match is next Wednesday's EIA WPSR release at 10:30 ET specifically. Most oil forecasts should be `EventHorizon(event_id="eia_wpsr", expected_ts_utc=...)`, not clock-horizon. Three things to specify:
   - Document when to use which horizon type (probably: `EventHorizon` for any forecast pinned to a scheduled release; `ClockHorizon` only for arbitrary-window backtests).
   - Specify the Forecast→Print match function explicitly. Best current form: match by `(target_variable, horizon.event_id if event else emission_ts + horizon.duration)` with a tolerance window.
   - Fix a policy for event-horizon events that slip (EIA delays release by a day): does the Forecast auto-re-anchor, or does the Grade fire on actual print time?

   Without these pinned, the grading harness will be written to whichever interpretation the first programmer assumes. Worth half an hour now.

## Significant — address in Week 0

4. **§8.2 CVaR-constrained portfolio sizing is under-specified.** The `combined_signal → position_sizes` step is load-bearing for attribution (position sizes feed the Grade/attribution chain) but §8.2 step 4 is one line with no:
   - Objective function (mean-CVaR, utility form, linear in signal)
   - Covariance source (who estimates, how often, what data)
   - Cold-start policy (new regime, no covariance history)
   - Position limits / sign constraints

   If CVaR sizing is genuinely Phase 1, it needs a §8.2a. Alternative: commit to linear sizing in Phase 1 (`position = clip(k × combined_signal, ±max)` with `k` regime-conditional) and defer CVaR to Phase 2. The capability claim doesn't require CVaR — it requires a deterministic, replayable, portable sizing function. Linear wins on all three. I'd cut CVaR from Phase 1.

5. **Weight matrix dimensionality is not addressed.** §8.1 says weights are `dict[regime, dict[(desk, target_variable), weight]]`. If desks emit multiple targets × multiple horizons, the matrix grows fast. Illustrative count: 6 desks × average 3 targets × average 2 horizons × 10 regimes = 360 weights. Fitting 360 weights on rolling windows with <5 years of daily data is overparameterised; most regimes will have <100 observations.

   Two disciplines worth pre-registering:
   - Cap the number of regimes the classifier emits at Phase 1 (maybe 4-6 in practice, even under HDP-HMM which nominally allows unbounded).
   - Weight-sharing across horizons within a (desk, target): weight is per (regime, desk, target), horizons share. Add a second weight for a per-horizon multiplier if needed. Cuts dimensionality roughly 2×.

   This is also an opportunity to make the weight matrix redeployable to equity VRP without dimensionality blow-up.

6. **Clean-code-commit policy for Provenance.** `code_commit: str` is fine when code is committed. Dirty working tree (uncommitted changes) is ambiguous — does the desk refuse to emit? Emit with `code_commit="<dirty>@<base_sha>"`? Crash? Needs a one-sentence policy. Recommended: emission with dirty tree is allowed in development mode, forbidden in production-mode replay runs, via a module-level flag. Log dirty emissions prominently in the audit trail.

7. **§6.2 is missing a data-ingestion-failure trigger.** EIA release delayed, OPEC MOMR format changed, CFTC site down — these are desk-independent but break the event-scoring loop. §14.5 acknowledges the risk but no trigger catches it. Add an 8th trigger:
   - `data_ingestion_failure` — fires when a scheduled ingestion didn't produce the expected Print by the expected time + tolerance. Priority 1. Payload identifies the affected feed. Research loop response: either wait, fall back to alternative source, or mark affected desks as stale.

   Without this, the only thing that fires when EIA is 24h late is a bunch of desks emitting `staleness=True` Forecasts, which is a diagnostic signal but not a loop trigger.

8. **Done-criterion "≥ 1 full round-trip per desk" is too light.** A single round-trip proves it *can* work, not that it *does* work. Strengthen to something like:
   - Event-scoring loop has run for ≥ 4 weeks continuously with no infrastructure incidents (gate-failures are not incidents; scheduler crashes or attribution-DB corruptions are).
   - Each desk has ≥ N closed round-trips (N = 10 is probably right for weekly-cadence desks; higher for daily).
   - Research-loop latency KPI is measurable and reported, not "pending data."

   The current bar of 1 round-trip lets the loop pass on a single happy-path run, which isn't the capability claim.

## Smaller — reasonable defaults, worth pinning

9. **§7.2 "N Controller revisions or M months" — unspecified N and M.** Pre-register or these become negotiated ad hoc. Suggest: M = 3 months, N = 2 Controller-weight promotions. Either triggers redundancy review.

10. **§10 classifier emits argmax regime, not distribution.** `RegimeLabel.regime_id: str` commits to a hard label. HDP-HMM naturally produces posteriors. Two options:
    - (a) Add `regime_probabilities: dict[str, float]` and let the Controller choose argmax vs weighted-average at decision time.
    - (b) Explicitly commit to argmax and treat `transition_probabilities` as the uncertainty channel.

    (a) is more expressive; (b) is simpler. Currently the schema implicitly commits to (b) without saying so.

11. **§6.4 routing rule "anything that writes to a desk spec → API" is only checkable ex post.** A task may start local and escalate into a spec change partway through. Reframe as: tasks execute local by default; any LLM output that includes a spec-change artefact must have been produced by an API call, or the artefact is rejected. This is a postcondition check, not a pre-routing rule.

12. **Macro desk vs regime classifier overlap (§5.3 + §10).** Macro emits regime-conditioning info as a Forecast. The classifier consumes desk Forecasts to emit a RegimeLabel. Under redeployment, Macro is replaced with an equity-macro desk; the classifier consumes its outputs. This is clean but worth one line in §10.1 confirming that "Macro desk output is valid classifier input by construction; Macro does not emit RegimeLabel directly."

13. **`attribution_lodo` table grain should probably be `(decision_id, desk_name)`, not `(print_id, desk_name)`.** LODO runs when a decision is made, not when a print lands. Multiple decisions may happen between prints. Schema fix is trivial but worth catching now — retrofitting attribution tables is painful.

## Missing from §14 — flagged risks

14. **Budget realism.** 6 months calendar assumes zero escalations past zero-shot on any desk. Probabilistically, with 6 desks each having some nonzero chance of hitting the classical-specialist escalation, expected number of escalations is likely ≥ 2. Each escalation adds 1–2 weeks of unplanned work. Realistic completion is 7–8 months; the 6-month budget is aspirational. Worth an explicit §14.6 noting this — not to relax the abandon rule but to pre-commit that exceeding 6 months without abandon-trigger is a capability-claim debit.

15. **Equity-VRP redeployment readiness assumption.** §14.2 notes the Phase 2 deadline is a capability-claim commitment, but doesn't flag the dependency: Phase 2 requires Speckle and Spot (or equivalent) to have desk candidates ready at Phase 1 exit. If the five equity-VRP desks don't exist in any form by month 6, Phase 2 can't start on time. Worth flagging that some Phase 2 prep must happen during Phase 1 (probably during Macro build, which is the lowest-intensity desk).

16. **Cold-start on Day 1 of live scoring.** The first time the event-scoring loop runs, no desk has a validation history, no regime classifier has fit, no weights exist. The Controller's fallback behaviour with empty weights isn't specified — does it emit no decision? Emit a zero-signal decision? Emit with the stub weight (uniform)? The degenerate-single-regime case (§10.3 hot-swap test) covers part of this but not the "literally no weights yet" case. Needs a policy.

## What's in good shape

- **§4.5 `test_boundary_purity.py`** is the right test to write Day 1. It's the single highest-leverage check you can put into the repo.
- **§12.3 abandon criteria** are sharp enough to force discipline. Tight, but intentionally so.
- The **LODO + Shapley co-primary framing (§9)** survived translation from the Q&A cleanly.
- **§15 derivation trace** is genuinely useful; keep the practice as v1.x revisions land.
- **`pyproject.toml`** is clean. One small addition worth making: `[tool.mypy] strict = true` scoped to `contracts/` (via `[[tool.mypy.overrides]]`). The contract module is where strict typing earns its keep.

## Recommendation

Before writing Week 0 code: fix items 1–3 (the blockers) as v1.1 revisions today, and take a view on 4 (CVaR vs linear sizing) before the bus is designed, since it affects the signature of the Controller's decision emission. Items 5–8 can be addressed as v1.1 over the first week. The rest are fine as v1.x over Phase 1.

**Single most important intervention**: cut CVaR from Phase 1 and commit to linear sizing (item 4). Every other complication downstream — covariance estimation, cold-start behaviour, portability — gets simpler. The capability claim doesn't need risk-optimal sizing; it needs deterministic portable sizing. Linear wins.
