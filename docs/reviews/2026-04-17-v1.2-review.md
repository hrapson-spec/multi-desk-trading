Captured verbatim below. Six items (A–F); all addressed in v1.3.

---

**A — §3.1 new invariant vs §14.8.** §3.1 asserts every Controller decision is immutable and attribution runs at decision time. §14.8 says Day-1 Controller operates under uniform weights emitted as ordinary `SignalWeight` rows with `validation_artefact="cold_start"`. These are consistent but one gap: a cold-start weight row uses `promotion_ts_utc = boot_ts`, which means multiple boot events in a development run produce multiple `promotion_ts_utc = boot_ts` rows per (regime, desk, target) triple. The §3.2 primary key on `signal_weights` is `(regime_id, desk_name, target_variable, promotion_ts_utc)`, so a same-second re-boot violates the PK. Low probability in practice but deterministic under test harnesses. Fix: cold-start rows carry an explicit bootstrap sequence counter, or `boot_ts` is specified to the microsecond.

**B — §8.2a indexing.** The sizing function stores `k_regime` and `pos_limit_regime` at `signal_weights[regime_id][(None, "__k__")]` and `signal_weights[regime_id][(None, "__pos_limit__")]`. Using `None` as the `desk_name` component and `"__k__"` as the `target_variable` works, but it means the `signal_weights` table's `desk_name` column must be nullable, and `target_variable` carries magic strings that are not in `KNOWN_TARGETS`. Two collisions with discipline committed elsewhere in the spec: `desk_name` as nullable weakens the type signature; `__k__` / `__pos_limit__` as `target_variables` violate §4.6's "every `target_variable` used in the system must be in the registry." The clean fix is a separate `controller_params` table with columns `(regime_id, promotion_ts_utc, k, pos_limit, validation_artefact)` — two additional columns, removes the magic-string ambiguity, preserves the registry invariant. Alternative: add `__k__` and `__pos_limit__` as sentinel `target_variables` in the registry with documentation that they are Controller-internal. Either works; the current formulation is ambiguous.

**C — §8.5 capacity arithmetic.** The "illustrative sizing" case quotes 108 weights + 12 = 120 parameters over ≥ 2 years, citing "≥ 200 observations per regime" as feasibility. Two things to check: the 200 figure presumes evenly-distributed regime occupancy, which HDP-HMM outputs rarely provide (typical regime occupancy is highly skewed — one regime holds 60% of observations, others 5–10%). Realistic minority-regime sample counts on 2 years of weekly data are sub-100 for the rarer regimes. Not a spec bug, but the illustrative feasibility claim is optimistic; worth a footnote that weight-fitting regularisation (ridge or Bayesian shrinkage) is assumed, not optional.

**D — §6.2 `data_ingestion_failure` payload.** "List of desks consuming that feed" is required in the payload. This means the bus or scheduler needs a data-source → desks mapping as a first-class registry. Not created by any scaffold artefact in `docs/plans/week0_scaffold.md` that I saw. Either add it explicitly to the scaffold (probably as `config/data_sources.yaml`), or weaken the payload requirement to "scheduled release that failed" and let the research loop join against desk specs at trigger time. Current state is a spec requirement without an owning artefact.

**E — §11 item 2 vs §8.3 step 6 rollback mechanism.** §11 item 2 says "any promotion event can be rolled back manually by writing a superseding `SignalWeight` row." §8.3 step 6 echoes this. Fine. But a superseding row is itself an auto-promotion candidate unless the pre-registered margin check is skipped. If a human writes a rollback row that does NOT beat the current matrix on held-out data (the whole point of rollback is often that the current matrix is misbehaving on recent data the validation didn't see), the promotion gate rejects it. Two ways out: rollback is a distinct operation (not via the auto-promotion path), requiring explicit human sign-off and a separate audit-log entry type; or the auto-promotion gate recognises a `rollback=True` provenance flag and bypasses margin check. The spec doesn't specify. Recommend the first — rollback as a distinct human action carries cleaner audit provenance. Worth one sentence in §8.3 or §11.

**F — §15 derivation trace entry for R3.** "Hybrid event-driven + periodic research loop" is attributed to R4 in the derivation trace. Checking the conversation history: the research-loop trigger pattern discussion was R6 in my count (after attribution methodology), hybrid landed there. Not load-bearing — the trace is for future readers to understand provenance, and "R4" vs "R6" doesn't affect the decision — but worth aligning with the actual round sequence if the trace is claimed as authoritative history.
